# Backend API Data Model Specification

This document outlines the SQLAlchemy relational mapping, SQLite database configurations, and stateless session mocks.

## 1. Database Connections & WAL Configuration

The database backend is initialized in [database.py](file:///home/rutvej/Desktop/DAA/app/backend-api/src/database.py). It reads `DAA_DB_PROVIDER` to determine the database engine:

- **PostgreSQL**: Used if `DAA_DB_PROVIDER` is set to `"postgres"`, `"internal-postgres"`, or `"external-postgres"`.
- **SQLite**: Used if `DAA_DB_PROVIDER` is set to `"sqlite"`. SQLite is configured with WAL (Write-Ahead Logging) mode to prevent concurrent write locking when multiple microservices submit logs:
  ```python
  @event.listens_for(engine, "connect")
  def set_sqlite_pragma(dbapi_connection, connection_record):
      cursor = dbapi_connection.cursor()
      cursor.execute("PRAGMA journal_mode=WAL")
      cursor.close()
  ```
- **Mock Database**: Activated if `DAA_DB_PROVIDER` is set to `"none"`, `"internal-redis"`, or `"external-redis"`.

## 2. Stateless Mode Mocks (`MockSession` and `MockQuery`)

When running in stateless serverless mode, local databases are disabled to allow the container to scale down to zero. The backend API uses `MockSession` and `MockQuery` classes to stub out SQLAlchemy operations:

### `MockQuery`
Stub class mocking query results:
- `filter()`, `filter_by()`, `join()`, `order_by()`, `limit()`, `offset()`: Return the `MockQuery` instance itself to allow method chaining.
- `first()`: Returns `None`.
- `all()`: Returns `[]`.
- `count()`: Returns `0`.

### `MockSession`
Stub class mocking transaction scopes:
- `query(model_class)`: Returns a new `MockQuery` instance.
- `add(instance)`: Intercepts models and assigns autogenerated UUIDs to the `id` field and the current UTC time to any datetime fields (`created_at`, `timestamp`, `first_seen_at`, `last_seen_at`).
- `delete()`, `commit()`, `rollback()`, `refresh()`, `close()`: Executed as no-ops.
- `begin()`: Returns a context manager stubbing a database transaction.

## 3. Database Table Definitions

SQLAlchemy models inherit from `declarative_base()`.
- **User**: Class `User` maps to table `users`.
- **Log**: Class `Log` maps to table `logs`. Links to `User` via foreign key `userId`.
- **Fix**: Class `Fix` maps to table `fixes`. Links to `Log` via foreign key `logId`.
- **ProjectConnection**: Class `ProjectConnection` maps to table `project_connections`.
- **Alert**: Class `Alert` maps to table `alerts`.
- **Application**: Class `Application` maps to table `applications`.
- **EscalationPolicy**: Class `EscalationPolicy` maps to table `escalation_policies`. Links to `Application` via foreign key `application_id`.
- **Incident**: Class `Incident` maps to table `incidents`.

## 4. Startup Migrations

During startup, the backend runs `run_db_migrations(engine)` in [database.py](file:///home/rutvej/Desktop/DAA/app/backend-api/src/database.py#L245-L260).
- It checks for the existence of the `applications` table.
- If columns `allowed_ip` or `token` are missing, it executes `ALTER TABLE applications ADD COLUMN ...` to update the schema dynamically.
