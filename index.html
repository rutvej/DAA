Overview

DAA (Debugging Autonomous Agent) is an open-source, self-hosted Debugging Autonomous Agent platform designed to automate distributed system debugging. Unlike passive alerting mechanisms, DAA conducts a structured investigation inside your codebase, queries environmental states, checks recent changes, and outputs a validated hotfix or structured postmortem reports.

Why DAA? Production incidents are rarely single-service errors. They cascade through queues, databases, and microservices. DAA connects the dots automatically by correlating distributed trace IDs, logs, metrics, and git histories.

Currently, the platform includes multi-language SDKs, a React SRE agent powered by LangChain, a central FastAPI backend, and an interactive admin dashboard supporting real-time incident status tracking.

Key Features in V3.0

DAA v3.0 brings critical SRE capabilities directly to the developer environment, optimizing for speed, accuracy, and low LLM token costs.

Zero Alert Fatigue

Sliding-window escalation thresholds and SHA256 error fingerprinting prevent duplicate alerts, wake-ups, and duplicate SRE tickets.

Three-Phase Hybrid Flow

Deterministic pre-flight (cloning/caching, telemetry log hydration, dedup) and post-flight (PR creation) wrapped around a free reasoning agent.

Context Safety System

Enforces a JSON planning step and an 8-call hard budget cap to prevent context explosion and costly agent hallucinations.

Universal LLM Routing

Support for Google Gemini, OpenAI, Claude, Vertex AI, Ollama, Codex, or agy proxy to easily change models without code edits.

Git & Ticket Automation

Creates GitHub Issues, GitLab/Gitea Pull Requests, and opens offline Jira tickets automatically when a verification fails.

Circuit Breakers

Halts operations, reports postmortems, and stops making commits if test suites fail twice or deadlocks are identified.

Human-in-the-Loop

Defers PR/MR creation for manual validation. Review postmortems and AI agent execution logs on the UI, then approve with a single click.

Model Context Protocol

Expose DAA SRE tools to other coding agents via a stdio MCP Server, or configure the DAA agent to consume external MCP tools.

Architecture & System Flow

DAA uses a decoupled, event-driven architecture designed to process, store, and analyze incident alerts without blocking production applications.

Services and Roles:

  • backend-api (app/backend-api): Manages Authentication, Ingests logs via REST API, matches deduplication metrics, and tracks fix progression.
  • python-agent (app/python-agent): Performs multi-dimension analysis using ReAct agent framework and pushes fixes to the Git repository.
  • admin-panel (app/admin-panel): React SPA displaying system configuration parameters, live logs, active incidents, and fix reports.
  • daa-sdk (app/daa-sdk): Multi-language clients enabling direct trace and exception transmission from services.

Workflow Lifecycle:

  1. An SDK error is triggered in the Client App.
  2. FastAPI stores the logs and verifies if the sliding-window threshold is breached.
  3. An IncidentEscalationEvent is queued via RabbitMQ.
  4. Phase 1: Pre-flight — Repo cache manager gets worktree, log hydrator fetches all 4 telemetry dimensions, and fingerprint dedup checks for existing fixes.
  5. Phase 2: Agent Core — SRE Agent submits a JSON plan first, then uses read-only tools to diagnose the bug and produce a unified diff under an 8-call safety cap.
  6. Phase 3: Post-flight — Post-flight orchestrator validates the diff, checkout/commits/pushes branch, opens PR idempotently, and generates a postmortem.

Quickstart & Setup Guide

1-Line Unified Installer (Linux/WSL/macOS)

The absolute fastest way to install DAA, configure environment variables, and build core libraries is using our unified automated curl installer script: 

curl -fsSL https://raw.githubusercontent.com/rutvej/DAA/main/install.sh | bash

Running the E2E Walkthrough Demo

DAA has an independent end-to-end sandbox walkthrough repository. Follow these steps to execute the automated resolution loop:

  1. Navigate to the Demo Walkthrough folder: 
    cd /home/rutvej/Desktop/daa-e2e-demo
  2. Execute the Demo Orchestrator Script: 
    python3 run_demo.py

    This spins up local Gitea services, invokes the DAA CLI (daa register and daa policy) to register applications/policies, triggers the Redis OOM, and lets the SRE agent solve it autonomously.

Redesigned SRE Workflow (Manual CLI Steps)

If you prefer to run the installation, initialization, registration, and policy setup manually to inspect the redesigned DAA SRE workflow, follow these steps:

  1. Initialize Demo Infrastructure & Repositories:

    Spin up Gitea, Redis, Postgres, and RabbitMQ, and push the local payment repositories to Gitea:

    cd /home/rutvej/Desktop/daa-e2e-demo
python3 run_demo.py --skip-registration --skip-load-test

    Take note of the Git Token generated and printed by the script (or retrieve it from .gitea_token).

  2. Install and Initialize DAA:

    Navigate to the DAA directory, run the unified installation script, link it globally, and run the configuration wizard:

    cd /home/rutvej/Desktop/DAA
./install.sh
sudo ln -sf $(pwd)/daa /usr/local/bin/daa
daa init
    • Select gitea as the Git provider.
    • Provide Gitea base URL http://localhost:3000 and the generated Git Token.
    • Provide your Gemini API key (e.g. AIzaSyYourGeminiApiKeyHere) and choose gemini-1.5-flash as the model.
    • Skip the cloud log connector configuration setup.
  3. Deploy DAA Services:

    Spin up the local backend, database, queue, and SRE agent services in Docker:

    daa redeploy
  4. Register Applications via DAA CLI:

    Register the microservices and obtain their telemetry tokens (you can run this from any folder now):

    daa register --name payment-api --repo-url http://host.docker.internal:3000/daa-admin/payment-api.git --language python
daa register --name payment-worker --repo-url http://host.docker.internal:3000/daa-admin/payment-worker.git --language go
  5. Configure Escalation Policies:

    Set up SRE escalation rules for the registered applications using the CLI:

    daa policy --app payment-api --threshold 3 --window 60
daa policy --app payment-worker --threshold 3 --window 60

    This triggers an SRE agent run if more than 3 errors are recorded within a 60-second window.

  6. Inject Telemetry Tokens & Deploy Microservices:

    Save the returned DAA_TOKEN values inside the demo's .env file (e.g. DAA_TOKEN_PAYMENT_API=<token>) and start the containers:

    # In /home/rutvej/Desktop/daa-e2e-demo/.env
cd /home/rutvej/Desktop/daa-e2e-demo
docker-compose up -d --build payment-api payment-worker
  7. Simulate the Outage:

    Flood the payment API cache to trigger the Redis OOM incident and launch the agent:

    ./load_test.sh

Human-in-the-Loop (HITL) Mode Setup

To enable safe approval gatekeeper logic, add DAA_HITL_MODE=true to your .env file. When an alert triggers the agent:

  • The agent will automatically execute the sequential 4-Dimension loop (Change, Infra, Logs, Diagnostics).
  • It will push the remediation code to a Git branch but defer PR/MR opening.
  • The Admin Panel will display a status of "Awaiting Approval" and expose full step-by-step AI Execution Logs.
  • Review the changes and click "Approve Fix & Create PR/MR" to trigger the Gitea/GitLab/GitHub REST API instantly.

Pluggable Cloud-Native Deployment

DAA features a fully pluggable, unified container architecture. The same Docker image (rutvej/daa:latest) can run in a zero-dependency stateless serverless mode, a self-contained single-container stateful VM mode, or a standard multi-container Docker Compose scale.

Stateless Serverless Recommendation: For Google Cloud Run or AWS Fargate deployments, configure DAA_DB_PROVIDER=none, DAA_GIT_MODE=api, and DAA_QUEUE_MODE=sync to run completely database-free, clone-free, and stateless.

Configuration Toggles

Variable Options Default Description
DAA_POLICY_ENABLED true, false false Enable/disable incident deduplication and cooldown checks.
DAA_AUTH_ENABLED true, false false Enable/disable JWT-based login portal authentication.
DAA_DB_PROVIDER none, sqlite, internal-postgres, external-postgres, internal-redis, external-redis sqlite Select database engine. Internal options start databases inside the container automatically.
DAA_GIT_MODE api, local local api reads and writes code directly via Git REST APIs (no cloning). local clones code.
DAA_QUEUE_MODE sync, rabbitmq sync sync processes jobs inline/background tasks. rabbitmq uses RabbitMQ queue.

Adoption Webhooks (SDK-Free Mode)

To bypass code changes or dependency installation, users can route webhooks directly from existing alert tools (Sentry, Prometheus, Datadog) to DAA. Webhooks are mapped either declaratively via YAML or using dynamic plugins: 

# Example: gcloud deploy one-liner for Cloud Run Webhook Receiver
gcloud run deploy daa-minimal \
  --image rutvej/daa:latest \
  --port 8080 \
  --set-env-vars="LLM_PROVIDER=google,GEMINI_API_KEY=xxx,GITHUB_TOKEN=xxx,DAA_DB_PROVIDER=none,DAA_GIT_MODE=api,DAA_QUEUE_MODE=sync"

Model Context Protocol (MCP) Integration

DAA v3.0 natively integrates with the **Model Context Protocol (MCP)** to support tool extensibility, cloud data integrations, and cross-agent collaboration.

A. Exposing DAA Tools to Other AI Agents (Server)

Allow other software engineering agents (such as Cursor, Claude Desktop, or custom coding copilots) to connect to DAA SRE using the stdio transport. This allows external agents to query pending fixes, read root-cause postmortems, or trigger PR approvals.

The DAA v3.0 MCP server exposes 7 powerful SRE tools:

  • get_fixes_awaiting_approval: Retrieve all DAA SRE incident remediation fixes currently awaiting human approval.
  • get_incident_postmortem: Get the detailed root-cause postmortem and AI agent trace for a given fix ID.
  • approve_remediation_fix: Approve a fix and trigger Gitea/GitLab/GitHub PR/MR creation.
  • get_active_incidents: Get all currently active (processing/pending) incidents with their app name, status, fingerprint, and elapsed time.
  • get_fix_by_fingerprint: Get fix record by error fingerprint hash. Returns existing PR URL if the same bug was previously fixed.
  • list_registered_apps: List all applications registered in DAA with their repository URL, cloud provider config, and escalation policy.
  • trigger_manual_incident: Manually trigger an incident analysis for a registered application by sending a synthetic error log to the DAA backend.

1. Running Natively on Host

Execute the server directly on your command line:

python3 app/daa_mcp_server.py

2. Running via Docker (Default Service)

The MCP server is configured as a Docker Compose service (mcp-server) by default. It starts automatically alongside other services:

docker-compose up -d

Other agents on your host can communicate with the containerized MCP server using standard Docker commands. For example, in your host's claude_desktop_config.json, register the server as follows:

{
  "mcpServers": {
    "daa-sre-mcp": {
      "command": "docker",
      "args": ["compose", "exec", "-T", "mcp-server", "python", "-u", "app/daa_mcp_server.py"]
    }
  }
}

B. Allowing DAA Agent to use External MCP Tools (Client)

Place an mcp_config.json file in the repository root. DAA will automatically launch any configured stdio-based MCP servers and append their tools to the agent's LangChain toolset during the diagnostic loop.

1. Database MCP Server Example

{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://youruser:password@localhost:5433/yourdb"]
    }
  }
}

2. Cloud & BigQuery MCP Server Example

To pull real-time analytics from GCP BigQuery or configure cloud-specific logging/APIs, configure the GCP credential environment paths in your mcp_config.json: 

{
  "mcpServers": {
    "bigquery": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-bigquery",
        "--project-id", "your-gcp-project-id"
      ],
      "env": {
        "GOOGLE_APPLICATION_CREDENTIALS": "/app/secrets/gcp-key.json"
      }
    }
  }
}

Ensure that your cloud keys/credentials are volume-mounted to your agent's container (e.g., /app/secrets/) in docker-compose.yml.

C. Automatic Preference: Choosing MCP over Direct REST APIs (Jira & Git)

If the DAA SRE Agent detects custom MCP tools are loaded for interacting with source control repositories or ticketing systems (such as GitHub/GitLab tools or Jira Cloud tools prefixed with mcp_), it will automatically choose the MCP tools instead of its direct local API tools (create_pull_request, create_incident_ticket, etc.).

This allows seamless integration of standard cloud-managed MCP services directly into the agent's reasoning loop without having to configure direct local environment variables for API tokens.

Platform Gaps & Roadmap

The SRE incident portal is constantly being updated. Below are the current active gaps and roadmap goals:

  • Three-Phase Hybrid Flow (Pre-flight, Agent Core, Post-flight orchestrators completed).
  • Context Safety System (JSON planning step and hard cap at 8 tool calls completed).
  • MCP Server v2.0.0 (Exposed 7 SRE tools for external agent execution completed).
  • Persistent Repo Cache (Cloning once, git-caching, and git worktrees per incident completed).
  • ChromaDB Repo Vector Search (RAG Indexing) (Semantic search over repo codebase to reduce file read token usage for DAA 3.1).
  • Real Cloud Log Connectors (Connectors for AWS CloudWatch, GCP Cloud Logging, and Datadog completed).
  • RabbitMQ Dead-Letter Queue (DLQ) (DLQ/DLX support for failed/crashed SRE agent jobs completed).
